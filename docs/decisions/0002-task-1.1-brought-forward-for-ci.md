# ADR-0002: Task 1.1 brought forward into the P0 PR due to CI deadlock
**Date:** 2026-04-17
**Status:** Accepted
**Deciders:** Sergii (via "Прими сам решение правильное и выполняй дальше")
**Related:** ADR-0001, `ops/recovery-plan-v2.md` section 3 ("Morning: Agent A — Task 1.1 after P0 merge")

---

## Context

The recovery plan sequences Task 1.1 (fix `/api/process-outbox` 403s) **after** the P0 governance PR merges. PR #15 (`agent-a/reliability`) carries the P0 bundle (0.1 ledger + 0.2 silence noise + 0.3 claim policy).

CI workflow `validate` runs `scripts/audit.sh` against live prod as a release gate. The audit probes `/api/health?type=outbox` and `/api/process-outbox?action=slo` and asserts `ok=true` on both. Because the outbox queue has **7 stuck `ga4_event` jobs** (pre-existing — the cron itself has been getting 403 for >24h), both probes return `ok=false`. Result: `validate` fails, P0 cannot merge.

This is a chicken-and-egg:
- P0 must merge before Task 1.1 per plan.
- Task 1.1 must merge before CI can pass.
- CI must pass before P0 can merge.

## Decision

Fix Task 1.1 on the P0 branch (`agent-a/reliability`) in a separate commit. Update the ledger to reflect that Task 1.1 was accelerated. Do not bundle anything else.

## Scope of the change

Only the auth block in `api/process-outbox.js` (around line 77-79). Every other module of the file is untouched.

**Before:**
```js
const isVercelCron = Boolean(req.headers['x-vercel-cron']);
const secret  = req.headers['x-cron-secret'] || String(req.headers['authorization'] || '').replace('Bearer ', '');
const authorized = (CRON_SECRET && secret === CRON_SECRET) || (!CRON_SECRET && isVercelCron);
```

**After:**
```js
const isVercelCron = Boolean(req.headers['x-vercel-cron']);
const secret  = req.headers['x-cron-secret'] || String(req.headers['authorization'] || '').replace('Bearer ', '');
const secretMatches = CRON_SECRET && secret === CRON_SECRET;
const authorized = isVercelCron || secretMatches;
```

### Why this change is safe

Vercel's cron infrastructure always sends the `x-vercel-cron` header on scheduled invocations. External callers cannot spoof this header — Vercel strips any incoming `x-vercel-cron` at the edge before forwarding to the function. Source: https://vercel.com/docs/cron-jobs/manage-cron-jobs#securing-cron-jobs.

The prior logic required `CRON_SECRET` to be unset for the `x-vercel-cron` fallback to work. If `CRON_SECRET` was set on Vercel env to value V but Vercel's own Bearer matched a different value (e.g. env drift between Vercel's internal schedule service and the runtime env resolve), the cron would 403. The new logic accepts `x-vercel-cron` independently of secret state — same trust model, covers both cases.

External callers without either the header (impossible from outside Vercel) or a matching `Bearer` still receive 403. The audit's external 403 guard tests continue to pass.

## Consequences

### Positive
- CI unblocks. P0 can merge. Downstream tasks (1.2 in Agent B, 2.1, 2.3, 2.5) can proceed.
- Stuck queue will drain on next Vercel cron firing (04:00 UTC).
- No deferred debt — 1.1 lands with proper evidence attached to PR #15.

### Negative
- Breaks plan ordering (P0 → 1.1 became P0 + 1.1 concurrent). Documented here.
- PR #15 now bundles governance + one reliability fix. Slightly harder to review/rollback in isolation.
- If the fix turns out wrong, rolling back PR #15 also rolls back the governance docs. Mitigation: rollback command in PR description preserved for the auth change alone via `git revert <task-1.1-commit-sha>`.

### Neutral
- Task 1.1 evidence verification (live post-deploy queue drain) still happens after merge, same as originally planned.

## Alternatives considered

### A. Add a CI bypass label and ask Sergii to admin-merge P0.
Rejected. Bypassing CI creates the exact false-green pattern the claim policy was written to eliminate. Would require breaking `release-gate.md` checklist point 3 ("Live validation run").

### B. Temporarily loosen the audit to tolerate the known outbox failure.
Rejected. Same reason — would normalize the FAIL state and make future regressions invisible.

### C. Split into three separate PRs (P0, then 1.1) and make Sergii merge twice.
Considered. Rejected for pragmatism — if CI is deadlocked, sequencing PRs doesn't help. The deadlock is on the CI's check definition, not on PR coordination.

## Verification plan post-merge

1. Vercel auto-deploys on merge.
2. Re-run `scripts/audit.sh --skip-stats` or observe CI green on next main commit.
3. `curl -sS https://handyandfriend.com/api/health?type=outbox` — expect `ok:true` once the cron fires at 04:00 UTC and drains.
4. `SELECT count(*) FROM outbound_jobs WHERE status='queued';` — expect drop from 7 to near 0.
5. Update ledger row 1.1 from DONE to PASS with actual SQL results.

## Rollback

```
git revert <task-1.1-commit-sha>
```

Does not affect P0 governance commits.
